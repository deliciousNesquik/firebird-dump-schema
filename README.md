# Firebird Dump Schema

Утилита командной строки для извлечения DDL-метаданных из баз данных Firebird в структурированную директорию (один объект — один файл). Предназначена для контроля версий схемы, резервного копирования и CI/CD.

Инструмент читает системный каталог напрямую через `firebird-lib` и запрашивает DDL у каждого объекта. Тип и имя объекта берутся из каталога, а системные объекты отфильтровываются автоматически по флагу каталога.

## Возможности

* **Дерево «один объект — один файл»** с устойчивыми путями — удобно для diff'ов и код-ревью схемы.
* **Автоматическая фильтрация системных объектов** по флагу каталога — без ручных списков.
* **Исполнимый вывод** — PSQL-объекты обёрнуты в блок `SET TERM`, каждый файл самодостаточен.
* **Безопасность** — учётные данные читаются из окружения, не передаются в аргументах команды и маскируются в логах.
* **Персистентный аудит** — весь вывод дублируется в `audit_YYYYMMDD.log`.

## Требования

* Python 3.11+.
* [`uv`](https://docs.astral.sh/uv/) (рекомендуется — собирает окружение из `pyproject.toml` автоматически).
* Клиентская библиотека Firebird (`libfbclient`), доступная для `firebird-driver`.
* Целевая СУБД: Firebird 3 / 4 / 5.

Зависимости (`firebird-driver`, `firebird-lib`, `python-dotenv`) и настройки `uv` (внутренний индекс, `native-tls`) объявлены в `pyproject.toml`; `uv` подтягивает их при запуске. Файл `uv.lock` фиксирует версии.

## Установка

```bash
git clone https://github.com/deliciousNesquik/firebird-dump-schema.git
cd firebird-dump-schema
cp .env.example .env   # затем заполнить учётные данные и пути
```

## Конфигурация (.env)

| Параметр | Описание |
| --- | --- |
| `ISC_USER` | Имя пользователя БД (например, SYSDBA). |
| `ISC_PASSWORD` | Пароль пользователя. |
| `FB_DATABASE` | Адрес БД. Локально — путь или алиас; удалённо — `HOST:ALIAS_OR_PATH`. |
| `DUMP_DIR` | Директория для сохранения структуры схемы. Абсолютный или относительный (от текущей рабочей директории) путь. |
| `ISQL_TIMEOUT` | Таймаут чтения метаданных в секундах. Значение `<= 0` отключает таймаут. POSIX-only (SIGALRM). |
| `DB_CHARSET` | Кодировка соединения (по умолчанию `UTF8`). Для legacy-баз с однобайтовыми метаданными укажите явно, напр. `WIN1251`. |
| `AUDIT_LOG` | Вести ли `audit_YYYYMMDD.log` (по умолчанию `true`). `false`/`0`/`no`/`off` — не создавать файл. |

## Использование

Три режима. Конфиг задаётся флагом `-c/--config` (по умолчанию `./.env`); имена объектов — позиционные аргументы.

```bash
# Полный дамп
uv run fb-dump-schema                      # .env
uv run fb-dump-schema -c production.env    # кастомный конфиг (несколько баз)

# Точечная выгрузка (когда заранее известно, какой объект нужен)
uv run fb-dump-schema ACCOUNT              # все совпадения по категориям
uv run fb-dump-schema ACCOUNT --type table # уточнить тип при совпадении имён
uv run fb-dump-schema CALC_TOTAL --stdout  # печать в консоль вместо записи в дерево
uv run fb-dump-schema V_REPORT --with-deps # объект + объекты, от которых он зависит

# Перечисление объектов (discovery)
uv run fb-dump-schema --list
uv run fb-dump-schema --list --type procedure

# Эквивалент без entry point
uv run python -m fbschema -c production.env
```

Относительные пути (`DUMP_DIR`, путь к `.env`) разрешаются от текущей рабочей директории.

**Точечный режим** по умолчанию пишет в то же дерево `DUMP_DIR`, обновляя только файлы названных объектов (дерево не очищается и не подчищается — удалённые объекты убирает полный дамп). Гранты и комментарии в точечном режиме не трогаются — их освежает полный дамп.

Флаг **`--with-deps`** дополнительно выгружает объекты, от которых зависят названные, транзитивно — чтобы получить самодостаточный набор для воссоздания. Источники: `RDB$DEPENDENCIES` (представления → таблицы, процедуры → процедуры/таблицы и т. п.) и домены столбцов/параметров (их Firebird в `RDB$DEPENDENCIES` не фиксирует). Не отслеживаются связи вне этих источников — например, таблицы по внешнему ключу или объекты из динамического SQL.

### Коды возврата

| Код | Значение |
| --- | --- |
| `0` | успех |
| `1` | ошибка инфраструктуры/конфига/таймаут |
| `2` | ошибка аргументов командной строки |
| `3` | частичный прогон: часть объектов пропущена (ошибка/нет прав) или запрошенные имена не найдены |

Сбой отдельного объекта логируется предупреждением и пропускается, не прерывая весь дамп (удобно для CI).

## Как работает `--with-deps`

Поиск зависимостей **рекурсивный (транзитивный)**: для каждого найденного объекта инструмент снова спрашивает «а от чего зависит он?» — и так до полного замыкания. Это обход в ширину (BFS): найденная зависимость кладётся в очередь и затем сама раскрывается.

Пример для представления `DBA$MONITOR`, которое читает таблицу `USERLIST`:

```
DBA$MONITOR (view)
├─ USERLIST (table)                      ← уровень 1: из RDB$DEPENDENCIES
│   ├─ BAS$ID, BAS$MEMO, BAS$SMALLINT,
│   │  BAS$VAR_20/250/255/32, …          ← уровень 2: домены столбцов USERLIST
├─ BAS$TIMESTAMP                         ← из тела представления (RDB$DEPENDENCIES)
└─ BAS$INTEGER, BAS$VAR_100              ← домены столбцов самого представления
```

Домены `BAS$ID`/`BAS$MEMO`/… столбцами самого `DBA$MONITOR` не используются — они подтянулись **через** `USERLIST`, то есть скрипт спустился на уровень глубже.

Что учитывается:

* **Глубина не ограничена** — цепочки вида `процедура → процедура → таблица → домен` обходятся целиком.
* **Защита от циклов** — взаимозависимые объекты (частое дело у PSQL-процедур) не зацикливаются: каждый объект обрабатывается один раз.
* **Два источника зависимостей**: `RDB$DEPENDENCIES` (представления → таблицы, процедуры → процедуры/таблицы, выражения и т. п.) **и** домены столбцов/параметров — связь «столбец → домен типа» Firebird в `RDB$DEPENDENCIES` не фиксирует, поэтому она собирается отдельно.
* **Не отслеживается**: связи вне этих источников — например, таблицы по внешнему ключу или объекты, упомянутые лишь в динамическом SQL.

## Структура вывода

В `DUMP_DIR` создаётся нумерованное дерево; имена файлов соответствуют именам объектов:

```
01_EXTERNAL_FUNCTIONS/   05_VIEWS/        09_PACKAGES/
02_GENERATORS/           06_EXCEPTIONS/   10_TRIGGERS/
03_DOMAINS/              07_FUNCTIONS/    11_ROLES/      (ROLES.sql)
04_TABLES/               08_PROCEDURES/   12_GRANTS/     (GRANTS.sql)
                                          13_COMMENTS/   (COMMENTS.sql)
```

* Для процедур и функций прямые объявления (forward declarations) складываются в `00_DECLARATION.sql`, а тела — в отдельные файлы по имени объекта (для разрешения циклических зависимостей).
* В файлы таблиц попадают `CREATE TABLE` и связанные констрейнты; индексы — отдельными файлами в `04_TABLES`.
* Генераторы по умолчанию выгружаются **без** текущего значения (это runtime-состояние, дающее шум в diff'ах при регулярном дампе); чтобы записать значение, добавьте флаг `--with-generator-values`.
* PSQL-объекты обёрнуты в блок `SET TERM ^ ;` и завершаются `^`; остальные — терминатором `;`. Каждый файл самодостаточен и исполним.

## Транзакции

Чтение метаданных всегда выполняется в транзакции **read-committed + record-version + NO WAIT** (обязательное правило инфраструктуры — WAIT может «подвесить» процесс на конфликте блокировок). NO WAIT принудительно навязывается транзакции, которую `firebird-lib` открывает внутри себя.

## Логирование

Диагностический вывод идёт в `stderr` (stdout остаётся чистыми данными — для `--list | grep` и `--stdout > file.sql`). В режимах записи в дерево (полный/точечный) он также дублируется в персистентный лог `audit_YYYYMMDD.log` в текущей рабочей директории; отключается через `AUDIT_LOG=false`. Пароль в логах маскируется (`********`).
