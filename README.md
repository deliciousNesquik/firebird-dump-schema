# Firebird Dump Schema

Инструментарий командной строки для автоматического извлечения DDL-метаданных из баз данных Firebird и последующего парсинга монолитного SQL-файла в структурированную директорию. Проект ориентирован на использование в CI/CD пайплайнах, системах резервного копирования и для контроля версий схем баз данных.

## Архитектура и преимущества

Проект имеет модульную структуру, что позволяет использовать его как единый конвейер или запускать компоненты независимо друг от друга.

* **Изоляция процессов:** Использование `awk` для высокопроизводительного парсинга DDL без создания узких мест по I/O.
* **Безопасность:** Учетные данные (ISC_USER, ISC_PASSWORD) передаются через переменные окружения, исключая их утечку через системные мониторы процессов (`ps aux`).
* **Гибкость:** Поддержка таймаутов, динамическое определение типов файловых систем для предотвращения долгих операций на сетевых дисках, настраиваемая фильтрация системных объектов.
* **Персистентное логирование:** Полный аудит выполнения с перехватом STDOUT и STDERR.

## Требования

* ОС на базе Linux (или WSL).
* Установленная утилита `isql` (входит в стандартную поставку Firebird).
* Стандартные утилиты GNU: `bash`, `awk`, `grep`, `xargs`.

## Установка

Клонируйте репозиторий и выдайте права на исполнение Bash-скриптам:

```bash
git clone https://github.com/deliciousNesquik/firebird-dump-schema.git
cd firebird-dump-schema
chmod +x *.sh

```

Создайте рабочий конфигурационный файл из предоставленного шаблона:

```bash
cp .env.example .env

```

## Конфигурация (.env)

Настройка параметров конвейера осуществляется через файл окружения.

| Параметр | Описание |
| --- | --- |
| `ISC_USER` | Имя пользователя базы данных (например, SYSDBA). |
| `ISC_PASSWORD` | Пароль пользователя. |
| `FB_DATABASE` | Путь к БД. Для локального подключения: путь или алиас. Для удаленного: `HOST:ALIAS_OR_PATH`. |
| `METADATA_FILE` | Имя или путь к временному файлу для монолитного дампа (например, `metadata.sql`). |
| `METADATA_FILE_DELETE_AFTER_PROCESSING` | Флаг (`true`/`false`) для автоматического удаления монолитного файла DDL после успешного парсинга. |
| `DUMP_DIR` | Директория, в которую будет сохранена структурированная схема БД. |
| `ISQL_PATH` | Абсолютный путь к исполняемому файлу `isql` (например, `/opt/firebird/bin/isql`, можно указать `isql` если установлен глобально). |
| `ISQL_TIMEOUT` | Время ожидания дампа в секундах. Значения меньше `1` отключают ограничение по времени. |

## Использование

Проект поддерживает два режима работы: полный автоматизированный цикл и точечный запуск утилит.

### Вариант 1: Запуск через оркестратор (`main.sh`)

Скрипт `main.sh` является точкой входа, которая последовательно выполняет валидацию конфигурации, извлечение схемы, парсинг файлов и очистку системы от встроенных объектов.

**Запуск со стандартным конфигурационным файлом (`.env`):**

```bash
./main.sh

```

**Запуск с кастомным конфигурационным файлом:**
Скрипт принимает путь к файлу окружения в качестве первого аргумента. Полезно для работы с несколькими базами данных.

```bash
./main.sh path/to/production.env

```

### Вариант 2: Использование независимых компонентов

Вы можете интегрировать отдельные скрипты в собственные процессы маршрутизации или автоматизации.

#### 1. Извлечение DDL-схемы (`fetch_schema.sh`)

Выполняет подключение к БД и формирует единый файл метаданных. Требует предварительно экспортированных переменных окружения.

```bash
# Пример ручного экспорта переменных
export ISQL_PATH="/opt/firebird/bin/isql"
export FB_DATABASE="localhost:/db/data.fdb"
export ISC_USER="sysdba"
export ISC_PASSWORD="masterkey"
export METADATA_FILE="raw_dump.sql"
export ISQL_TIMEOUT=300

./fetch_schema.sh

```

#### 2. Разделение объектов (`split_objects.sh`)

Принимает готовый DDL-файл и разбивает его по директориям в зависимости от типов объектов (процедуры, триггеры, таблицы и т.д.).
**Синтаксис:** `./split_objects.sh <путь_к_ddl_файлу> <целевая_директория>`

```bash
./split_objects.sh "raw_dump.sql" "parsed_schema_dir"

```

#### 3. Очистка системных объектов (`clean_system.sh`)

Удаляет из целевой директории системные функции и процедуры, которые не должны попадать в систему контроля версий (списки для удаления хранятся в файлах `*.list`, например `system_procedures.list`).
**Синтаксис:** `./clean_system.sh <целевая_директория>`

```bash
./clean_system.sh "parsed_schema_dir"

```

## Логирование

Скрипты используют встроенную библиотеку `utils.sh` для форматированного вывода. При запуске через `main.sh` весь поток (STDOUT и STDERR) дублируется в персистентный лог-файл вида `audit_YYYYMMDD.log` в рабочей директории.
